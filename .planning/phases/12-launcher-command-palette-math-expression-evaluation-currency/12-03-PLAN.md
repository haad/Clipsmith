    - Static query regex for shape detection (RESEARCH.md Pattern 4 — same regex as UnitConversionService but matched at the orchestration layer so we can branch on "currency-shaped" vs "unit-shaped" before calling either service):

      ```
      private static let conversionRegex = try! NSRegularExpression(
          pattern: #"^([\d.,]+(?:[eE][+-]?\d+)?)\s+([\w°]+)\s+(?:to|in)\s+([\w°]+)\s*$"#,
          options: [.caseInsensitive]
      )
      ```

    `func evaluate(_ query: String) -> CommandResult?`:
    1. `let trimmed = query.trimmingCharacters(in: .whitespaces); guard !trimmed.isEmpty else { return nil }`
    2. Try conversion shape first by running `conversionRegex.firstMatch(in: trimmed, range: ...)`. If match:
       - Extract captures: value string (group 1), fromToken (group 2), toToken (group 3)
       - `if CurrencyService.isCurrencyQuery(from: fromToken, to: toToken)` → call `currencyService?.convert(amount: amount, from: fromToken, to: toToken)`; on non-nil result return CommandResult(kind: .currency, displayValue: ExpressionEvaluator.formatResult(value), copyableValue: ExpressionEvaluator.copyableResult(value), expression: trimmed, toUnit: toToken.uppercased()). On nil → return nil (no fallthrough; user intent was currency conversion and we couldn't satisfy it).
       - Else fall through to UnitConversionService: `if let (value, unit) = UnitConversionService.convert(trimmed)` → return CommandResult(kind: .unit, displayValue: ExpressionEvaluator.formatResult(value), copyableValue: ExpressionEvaluator.copyableResult(value), expression: trimmed, toUnit: unit). On nil → return nil (we matched the conversion shape but couldn't convert).
    3. If conversion shape did NOT match, fall through to bare math: `if let value = ExpressionEvaluator.evaluate(trimmed)` → return CommandResult(kind: .math, ..., toUnit: nil). On nil → return nil.

    Critical ordering reasons (document in doc comment):
    - Currency check FIRST because "5 USD to EUR" matches the conversion regex AND its tokens are all 3-letter ISO codes. If we ran UnitConversionService first, its disambiguator (added in Plan 02) would return nil and we'd then have to retry — wasteful and confusing.
    - Math LAST because expressions like "2+2" never contain " to " or " in " — they fail the conversion regex up front and only ExpressionEvaluator can handle them.
