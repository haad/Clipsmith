# Phase 03.1: ObjC Parity Bug Fixes - Research

**Researched:** 2026-03-06
**Domain:** Swift/macOS — clipboard manager parity fixes across data layer, hotkey UX, bezel UI, and settings
**Confidence:** HIGH — all findings from direct codebase inspection; no speculative claims

---

## Summary

Phase 03.1 is a surgical parity sprint: the Swift rewrite's architecture is sound but ~30 behavioural
gaps remain between it and the original ObjC Flycut. A detailed fix plan already exists in
`docs/PARITY-BUGFIX-PLAN.md` — this research document maps that plan onto the actual Swift source
code, confirms each fix's pre-conditions and exact call sites, and surfaces cross-cutting concerns
that affect plan structure.

The fixes are pre-organised into six dependency-ordered waves by the bug plan. The waves map cleanly
onto separate plans: Wave 1 and 2 are the most structurally significant (they change callback
signatures and add new behaviour to BezelController); Waves 3–6 are largely additive (new key
handlers, settings, UI polish). The test infrastructure (XCTest + in-memory SwiftData containers)
is already in place and comprehensive — test updates are straightforward for every fix.

**Primary recommendation:** Follow the six-wave order from the bug plan without deviation. Waves 1
and 2 must be atomic units (cross-file signature changes). Waves 3–6 can each be a separate plan
or bundled as pairs if preferred.

### Pattern 1: Callback Signature Change (Wave 1 — Bug #1)

**What:** `ClipboardMonitor.onNewClipping` changes from `((String)->Void)?` to `((ClipboardEntry)->Void)?`
**Impact radius:** 3 files must change atomically — `ClipboardMonitor.swift`, `AppDelegate.swift`, `ClipboardMonitorTests.swift`
**Key constraint:** `ClipboardEntry` struct must be defined inside `ClipboardMonitor.swift` (or a shared file) — it must be `Sendable` since it crosses actor boundaries in `AppDelegate`

```swift
// ClipboardMonitor.swift — new nested struct
struct ClipboardEntry: Sendable {
    let content: String
    let sourceAppName: String?
    let sourceAppBundleURL: String?
}
var onNewClipping: ((ClipboardEntry) -> Void)?

// checkPasteboard() — add after content extraction:
let frontApp = NSWorkspace.shared.frontmostApplication
let entry = ClipboardEntry(
    content: content,
    sourceAppName: frontApp?.localizedName,
    sourceAppBundleURL: frontApp?.bundleURL?.path
)
onNewClipping?(entry)
```

**Timing note:** `NSWorkspace.shared.frontmostApplication` at paste time may return Flycut itself
briefly. This is acceptable — AppTracker already has this race condition and logs it. The clipboard
monitor fires on a 0.5s timer, so by the time `checkPasteboard()` runs the user's app is still
frontmost in almost all cases.

### Pattern 2: ClipboardStore insert() Signature Change (Wave 1 — Bug #1 + #2)

**What:** `insert(content:rememberNum:)` → `insert(content:sourceAppName:sourceAppBundleURL:rememberNum:)`
**Impact radius:** `ClipboardStore.swift`, `AppDelegate.swift`, `ClipboardStoreTests.swift`
**Dedup behaviour change (Bug #2):** Current code: `guard duplicate == 0 else { return }` → New code: update timestamp + metadata, early return without inserting duplicate object

```swift
// ClipboardStore.swift — new insert() body
if let duplicate = existing.first {
    duplicate.timestamp = .now
    duplicate.sourceAppName = sourceAppName
    duplicate.sourceAppBundleURL = sourceAppBundleURL
    try modelContext.save()
    return
}
// ... normal insert path unchanged
```

**Test update:** `testDuplicateSkipped` in `ClipboardStoreTests` — rename to `testDuplicateMovedToTop`
and verify count stays 1 AND re-fetched item is still the most-recent (timestamp = .now).

### Pattern 3: Hotkey Hold-Navigate-Release (Wave 2 — Bug #3)

**Critical architectural decision:** `flagsChanged(with:)` override on NSPanel.

NSPanel with `.nonactivatingPanel` does NOT own the keyboard event stream in the normal sense — it
becomes key but the app does not activate. `keyDown` events work because `canBecomeKey = true` and
`makeKeyAndOrderFront` is called. However `flagsChanged` is an event type that may or may not reach
the panel depending on macOS version and focus state.

**Verified approach (from bug plan):** Use the global monitor fallback FIRST (not as fallback):

```swift
// BezelController.swift
var isHotkeyHold = false
private var flagsMonitor: Any?

private func registerFlagsMonitor() {
    flagsMonitor = NSEvent.addGlobalMonitorForEvents(matching: .flagsChanged) { [weak self] event in
        guard let self, self.isHotkeyHold else { return }
        let modifiers: NSEvent.ModifierFlags = [.command, .option, .control, .shift]
        if event.modifierFlags.intersection(modifiers).isEmpty {
            Task { @MainActor in
                await self.pasteAndHide()
                self.isHotkeyHold = false
            }
        }
    }
}

private func removeFlagsMonitor() {
    if let m = flagsMonitor { NSEvent.removeMonitor(m); flagsMonitor = nil }
}
```

Register in `show()`, remove in `hide()`. This mirrors the existing click-outside pattern in
`registerClickOutsideMonitor()` — the code pattern is already established in `BezelController`.

**AppDelegate hotkey handler change (Bug #3):**

```swift
// Replace the existing KeyboardShortcuts.onKeyDown(for: .activateBezel) block
KeyboardShortcuts.onKeyDown(for: .activateBezel) { [weak self] in
    Task { @MainActor in
        guard let self else { return }
        let stickyBezel = UserDefaults.standard.bool(forKey: "stickyBezel")
        if self.bezelController.isVisible {
            self.bezelController.viewModel.navigateDown()
        } else {
            self.bezelController.isHotkeyHold = !stickyBezel
            self.bezelController.show()
        }
    }
}
```

Note: the current handler already calls `self.bezelController.viewModel.navigateDown()` when the
bezel is visible — this is already partially correct. Only `isHotkeyHold` and `stickyBezel` are new.

### Pattern 4: BezelViewModel Data Model Enrichment (Wave 3/4 — Bugs #11, #14)

**Current:** `BezelViewModel.clippings: [String]` — content only.
**Required for Bug #14 (source display):** Clippings need metadata alongside content.

**Recommended approach:** Introduce a lightweight `ClippingInfo` struct in `BezelViewModel.swift`:

```swift
struct ClippingInfo: Sendable {
    let id: PersistentIdentifier
    let content: String
    let sourceAppName: String?
    let sourceAppBundleURL: String?
    let timestamp: Date
}
```

`BezelView.updateClippings()` currently does `viewModel.clippings = clippings.map(\.content)`.
Change to map to `[ClippingInfo]`. This also enables Bug #11's delete-from-bezel (needs the
`PersistentIdentifier` to delete) and Bug #23's pasteMovesToTop (needs content string, already
available).

**Test impact:** `BezelViewModelTests` is entirely `[String]`-based — update `makeViewModel` helper
and all tests to use `ClippingInfo`. Tests remain pure-Swift (no SwiftData dependency) since
`PersistentIdentifier` can be created in tests from the test container.

### Pattern 6: BezelController ClipboardStore Injection (Wave 3 — Bug #23)

`BezelController` currently has no reference to `ClipboardStore`. Bug #23 (`pasteMovesToTop`) and

**Add to `BezelController.swift`:**
```swift

```

**Wire in `AppDelegate.applicationDidFinishLaunching`** (after `bezelController` init):
```swift

```

This is the same pattern used for `pasteService` and `appTracker`.

### Pattern 7: BezelController `pasteAndHide()` Visibility

`pasteAndHide()` is currently `private`. Bugs #13 (double-click) and Wave 3 mouse handling require
calling it from closures or gesture callbacks. Change visibility to `internal` (remove `private`).

### Anti-Patterns to Avoid

- **Do NOT use `flagsChanged` override alone** — non-activating panels do not reliably receive
  `flagsChanged` events on all macOS versions. Use `NSEvent.addGlobalMonitorForEvents` (the same
  pattern as click-outside monitoring, already established in the codebase).
- **Do NOT change `BezelViewModel.clippings` to `[FlycutSchemaV1.Clipping]`** — `@Model` objects
  are not `Sendable`. Keep ViewModel as pure Swift structs. The `BezelView.updateClippings()` bridge
  is the right abstraction layer.
- **Do NOT call `NSApp.activate()` in `pasteAndHide()`** — this would steal focus from the target
  app before paste. `PasteService.paste()` already handles app activation correctly.
- **Do NOT skip updating `ClipboardStoreTests.testDuplicateSkipped`** — its assertion (`count == 1`)
  still passes after the Bug #2 fix, but the test name and intent are wrong. A separate
  `testDuplicateMovedToTop` test is required.

## Common Pitfalls

### Pitfall 1: `flagsChanged` Unreliability on Non-Activating Panels

**What goes wrong:** Override `flagsChanged(with:)` on `BezelController` (NSPanel). On macOS 14+
with Stage Manager or certain focus states, the event may not arrive even though the panel is key.
**Why it happens:** Non-activating panels sit outside the normal responder chain for modifier events.
**How to avoid:** Use `NSEvent.addGlobalMonitorForEvents(matching: .flagsChanged)` registered in
`show()` and removed in `hide()`. This is already the established click-outside pattern.
**Warning signs:** `flagsChanged` tests pass in unit tests (no display server) but fail during
manual testing.

### Pitfall 2: `NSWorkspace.frontmostApplication` Returns Flycut at Copy Time

**What goes wrong:** If clipboard check fires very shortly after Flycut bezel closes or hotkey
fires, `frontmostApplication` may momentarily return Flycut's own process.
**Why it happens:** App activation is asynchronous; the 0.5s poll timer may catch a transition state.
**How to avoid:** In `ClipboardMonitor.checkPasteboard()`, compare `frontApp?.bundleIdentifier`
against `Bundle.main.bundleIdentifier` and leave `sourceAppName`/`sourceAppBundleURL` as `nil`
rather than storing "Flycut" as the source. This matches ObjC behaviour.
**Warning signs:** Clippings in the bezel show "Flycut" as their source application.

### Pitfall 3: `ClipboardStore.insert()` Signature Change Breaks Existing Tests

**What goes wrong:** `ClipboardStoreTests` calls `store.insert(content:rememberNum:)` — after the
Wave 1 signature change to `insert(content:sourceAppName:sourceAppBundleURL:rememberNum:)`, all
existing tests fail to compile.
**How to avoid:** Update all call sites in tests to pass `sourceAppName: nil, sourceAppBundleURL: nil`
OR add a convenience overload that defaults those to `nil`. Recommended: use `nil` parameters in
tests rather than adding an overload (keeps the API surface clean).

### Pitfall 4: `BezelViewModel.clippings` Type Change Breaks `BezelViewModelTests`

**What goes wrong:** All 19 `BezelViewModelTests` use `vm.clippings = ["a", "b", "c"]`
(string array). After changing to `[ClippingInfo]`, every test must be updated.
**How to avoid:** Add a test-only helper `makeClippingInfos(_ strings: [String]) -> [ClippingInfo]`
in a test file that creates stub `ClippingInfo` values with nil metadata. Centralises the update.
**Warning signs:** Mass test compilation failures in `BezelViewModelTests` after Wave 3+ changes.

### Pitfall 5: `savePreference == 0` (Never Save) Race on Quit

**What goes wrong:** `applicationWillTerminate` fires `Task { try? await clipboardStore.clearAll() }`
but the task may not execute before the process exits if the system terminates immediately.
**Why it happens:** Swift concurrency tasks are async — `applicationWillTerminate` returns before the
task completes.
**How to avoid:** For the "never save" mode, use a synchronous SwiftData delete on the main context
directly (via `FlycutApp.sharedModelContainer.mainContext`) rather than going through the async
actor. This matches how `MenuBarView` already does direct deletes.

### Pitfall 6: `MenuBarExtra(.menu)` Does Not Support `TextField`

**What goes wrong:** Bug #21 proposes adding a search TextField inside the `.menu` style
`MenuBarExtra`. SwiftUI's `.menu` style renders as an `NSMenu` which does not support hosted
`TextField` controls.
**How to avoid:** Implement Bug #21 only if switching to `.window` style, OR simply note that
in-bezel search (already wired to the search hotkey) covers this use case. The bug plan already
flags this caveat — defer to a later phase or skip for parity purposes.
**Warning signs:** TextField appears in the menu but receives no keyboard input.

### Pitfall 7: `isHotkeyHold` Not Reset on All Exit Paths

**What goes wrong:** If the bezel is hidden via Escape, Enter, or click-outside while `isHotkeyHold`
is true, the flags monitor fires on the next modifier release and calls `pasteAndHide()` even
though the bezel is already hidden.
**How to avoid:** `hide()` MUST set `isHotkeyHold = false` AND call `removeFlagsMonitor()`.
The current `hide()` does neither — these must be added in Wave 2. Verify in BezelControllerTests.

## Code Examples

Verified patterns from existing codebase (all sources are direct file reads):

### Existing Global Monitor Pattern (click-outside — `BezelController.swift`)

```swift
// Source: FlycutSwift/Views/BezelController.swift lines 195-211
private func registerClickOutsideMonitor() {
    removeClickOutsideMonitor()
    globalMonitor = NSEvent.addGlobalMonitorForEvents(matching: .leftMouseDown) { [weak self] event in
        guard let self else { return }
        if !self.frame.contains(NSEvent.mouseLocation) {
            Task { @MainActor in self.hide() }
        }
    }
}

private func removeClickOutsideMonitor() {
    if let monitor = globalMonitor {
        NSEvent.removeMonitor(monitor)
        globalMonitor = nil
    }
}
```

Apply the **exact same pattern** for `flagsChanged` monitor in Wave 2.

## Open Questions

1. **`flagsChanged` on non-activating panel — macOS 15 behaviour**
   - What we know: The bug plan already identifies this risk and recommends the global monitor fallback
   - What's unclear: Whether the panel-level `flagsChanged` override works at all on macOS 15 Sequoia
   - Recommendation: Skip the panel-level override entirely; use ONLY the global monitor pattern
     (consistent with the click-outside approach, safer, already proven to work)

2. **`BezelViewModel` type change scope**
   - What we know: Changing `clippings: [String]` to `clippings: [ClippingInfo]` requires updating
     ALL 19 `BezelViewModelTests` and `BezelView.updateClippings()`
   - What's unclear: Whether ClippingInfo should be defined in BezelViewModel.swift or a shared file
   - Recommendation: Define `ClippingInfo` in `BezelViewModel.swift` — it is a pure view-layer struct
     with no SwiftData dependency

3. **`removeDuplicates` setting interaction with Bug #2**
   - What we know: Bug #7 adds a `removeDuplicates: Bool = true` setting; Bug #2 changes dedup
     behaviour from "drop" to "move-to-top"
   - What's unclear: When `removeDuplicates = false`, should re-copying insert a second copy or still
     move-to-top? ObjC behaviour: dedup is bypassed entirely → two separate entries exist
   - Recommendation: When `removeDuplicates = false`, skip the duplicate check in `ClipboardStore.insert()`
     entirely and always insert a new entry

4. **`menuSelectionPastes` toggle semantics**
   - What we know: ObjC has this setting; when false, clicking a menu item copies to clipboard
     without pasting
   - What's unclear: The current `MenuBarView` button action calls `pasteService.paste()` directly
   - Recommendation: In Wave 5, check the setting in the menu button action: if false, write to
     clipboard but do not call `pasteService.paste()`
